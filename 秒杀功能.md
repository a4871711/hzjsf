# 秒杀功能 · 需求与实现计划书

> 来源：`秒杀(1).docx`（含 6 张标注截图）＋ 你已写的初稿。
> 现状勘察日期：2026-07-16（对照仓库当前 `main` 分支实际代码 + 一轮跨 5 个商品域的并行代码调研）。
> 一句话目标：把后台营销后台升级为「营销中心 → 秒杀管理」，让运营能对**任意后台可售商品**配置限时秒杀；会员端 uniapp 首页出现秒杀卡片、可按秒杀价抢购。

## 0. 结论速览（先看这段）

- **Q1 已拍板**：秒杀商品池 = **所有后台创建出来的可售商品**（会员卡 / VIP权益卡 / 私教商品 / 商城商品 / 团课），不局限于私教商品。这把数据模型从"绑一张 `pt_product`"变成**多态绑定 `(业务类型, 商品ID)`**。
- **秒杀后端骨架已存在**，但它是**照私教商品 `pt_product` 单一绑定**建的：表 `mk_flash_sale_activity`、后台接口 `/sys/mkFlashSale/*`、admin 页 `marketingFlashSale.vue`。**私教域是全仓库唯一已完整接好秒杀的域，作为其它域的参考范式（事实标准）。**
- **要做的三件大事**：① 把秒杀活动模型改成**多态**、后台创建页按 docx 扩字段；② 给其余 4 个商品域各接一套"秒杀价覆盖 + 秒杀库存扣减 + 限购校验"，并让会员端"立即抢购"**按商品类型分发到各自现有的下单链路**；③ 会员端 uniapp 从零做秒杀卡片 + 抢购交互。
- **"团购价"在所有域都不存在**（会员卡/私教/权益卡/团课/商城都没有"团购价"字段）。docx 的"团购价 / 售罄恢复团购价"需你明确映射到各域哪个价（原价划线基准），见 §5-Q1a。
- **难度极不均衡**：私教（已接好）＜ 权益卡·会员卡普通（价格后端可控、改动集中）＜ 会员卡自动续费·团课（有坑）≪ 商城（价格全前端可控、无库存扣减、会员端零 UI，最深坑）。建议分期见 §6。

> ✅ **已锁定决策（2026-07-16）**：
> 1. **P0 只接 3 个域**：私教商品 + VIP权益卡 + 会员卡·普通卡（自动续费卡/团课/商城押后）。
> 2. **价格口径**：统一取各域"对外售价"作**划线原价**，**秒杀价即成交价**；不新增"团购价"字段（docx 文案"团购价"= 对外售价）。
> 3. **P0 不补服务端价格防篡改**：秒杀价由服务端按活动写入订单、前端传一致值即可；**接受**会员卡等既有"信任前端价格"缺陷带来的薅羊毛风险（留待 P2 统一整改）。
> 4. **P0 只做单次生效 + 总量库存**（循环生效/每日时段/倒计时预热/售罄双策略放 P1）。

## 1. 需求背景与范围界定

给会员端做**限时秒杀**营销位：运营在后台配置"某商品在某段时间以更低的秒杀价、限量出售"，会员在小程序首页显眼位置看到倒计时卡片并抢购。同时把零散的营销后台（优惠券/拼团/秒杀）统一收进「营销中心」。

**范围裁剪（docx 正文明确）**：
- **不分新老客**——docx 正文「创建秒杀（**不分新老**，分单用户购买份数）」。参考图里"生效范围：所有人/店铺新客"**砍掉店铺新客**，固定"所有人"；只保留"单用户单品限购份数"。
- 覆盖：后台创建/管理 + 会员端展示/抢购两条主线；不含美团/抖音第三方侧活动配置（仅留团购ID字段用于关联/核销）。

## 2. 商品池与多态接入（本次核心复杂度，Q1 的落地）

### 2.1 五个可售商品域全景（已穷举，无遗漏）

> 查漏结论：储值充值/钱包、私教分期、优惠券、拼团都是"支付方式"或"营销活动"，不是可被秒杀的基础商品品类。基础可售品类就是下面 5 个。

| 商品域 | 商品表 | 会员端下单接口 | 支付回调分发 | 订单表 |
|---|---|---|---|---|
| **私教商品** | `pt_product` | `POST /api/privateOrder/create`（含 `quote` 试算） | `proPayNotify` 后缀 `b` | `pt_private_order` |
| **会员卡** | `fit_card`（+`device`资产） | `POST /api/cardOrder/createOrder` | `proPayNotify` `CARD_ORDER_TYPE` | `card_order` |
| **VIP权益卡** | `vip_benefit_card`（+`vip_benefit`实例） | `POST /api/vipCard/buy` | `proPayNotify` 后缀 `6` | `vip_benefit`（无独立订单表） |
| **团课** | `team_class`（单场次） | `POST /api/teamClassOrder/createOrder` | `PayController` `TEAM_CLASS_ORDER_TYPE` | `team_class_order` |
| **商城商品** | `goods`（+`order_detail`） | `POST /api/goodsOrder/addGoodsOrder` | `wxNotify/zfbNotify` 后缀 `3` | `goods_order` |

### 2.2 各域秒杀接入难度矩阵（决定分期与工作量）

| 域 | 秒杀价覆盖点 | 价格是否可信 | 库存现状 | 原生限购 | 主要坑 | 难度 |
|---|---|---|---|---|---|---|
| **私教商品** | `PrivateOrderServiceImpl.applyActivity()` **已接** `marketingType=2`+`flash_sale_price` | ✅ 后端重算，quote/create 同价 | 商品 `sale_stock` + 活动 `activity_stock`，回调 **CAS 原子扣两套**防超卖 | 商品级+活动级双重，**已实现** | 会员端 UI 未建 | ✅ **已是范式** |
| **权益卡** | `VipBenefitServiceImpl.buy():63` `currentPrice` 一处 | ✅ 后端算，前端不传价 | 无库存字段；`sold_count` 只用于**动态涨价**与展示 | "持有效即禁购"（一人一卡） | 动态涨价价 vs 秒杀价如何取舍；`status=9` 待支付占位单**无超时回收** | 🟡 中，改动集中 |
| **会员卡（普通）** | `CardOredrController.createFitCardOrder:110-115` | ⚠️ **信任前端 paySum 缺陷**，秒杀价须服务端权威写 `card_order.paySum` | 无库存字段，秒杀库存全由活动表维护（比私教更干净、无双库存冲突） | 无原生限购 | `cardNature=1` 权益门槛、次卡类型互斥不能被秒杀跳过 | 🟡 中 |
| **会员卡（委托代扣自动续费）** | 同上 + 须绕过续费分支 `162-181` 的 `nextPrice` | ⚠️ 同上 | 无库存 | 无 | **首期秒杀价扣完，`AutoPayTask` 后续每月仍按 `nextPrice` 原价扣**——秒杀是否只作用首期？ | 🟠 偏高 |
| **团课** | `TeamClassOrderController.createOrder:52-54` | ⚠️ 信任前端 `money`（`/api/pay/rechargePay`）缺陷 | `totalNum`/`actulNum` 单场次容量，回调 `+1` **非原子、已存在超卖** | 必须持有效卡会员 + 同课判重 | 是**单场次约课非可复用 SKU**；线上三表 **0 行、功能休眠**；"必须持卡"与秒杀拉新冲突 | 🟠 偏高 |
| **商城商品** | ❌ **无服务端价格权威**（`addGoodsOrder` 原样落前端传入的 `realPayment`/`goodsPrice`） | ❌ **价格完全前端可控**，后端只校验 `realPayment>=0` | `goods.total` **下单/支付全程从不扣**、无"已售" | 无 | uniapp **零商城 UI**（2018 遗留、仅后端接口+旧后台）；实物**需发货**；`goods.price` 是整数 | 🔴 **最深坑** |

### 2.3 多态数据模型（把秒杀活动从"绑私教"改成"绑任意商品"）

现状 `mk_flash_sale_activity.product_id` 硬绑 `pt_product`（`SysMarketingFlashSaleServiceImpl` 里用 `ptProductDao` 校验商品必须上架）。改造方向：

- 活动表增加 **`biz_type`（业务类型：私教/会员卡/权益卡/团课/商城）** + 沿用 `product_id` 作为该类型下的商品ID → 组成多态引用 `(biz_type, product_id)`。
- 后台创建页的"选择商品"先选**商品类型**、再从该类型的商品列表选具体商品；秒杀价 ≤ 该商品对外售价的校验按类型取价。
- 会员端"立即抢购"按 `biz_type` **分发到对应域的 `createOrder`**（见 2.1 表），传入秒杀活动ID + 秒杀价；各域回调按各自后缀走既有分发，只需在各域下单/回调里补"秒杀价覆盖 + 秒杀库存 CAS 扣减 + 限购校验"三件套（私教已有，可直接抄范式）。
- **秒杀库存统一由活动表承载**：除私教外各域商品本身多无库存概念，秒杀"限量 N 份"就是活动表的 `activity_stock`/`sold_count`，用 `UPDATE ... SET sold_count=sold_count+1 WHERE sold_count<activity_stock` 的 CAS 防超卖（照私教 `increaseFlashSaleSoldCount`）。

## 5. 待确认的产品决策点（动手前逐条拍板；已按重要性排序）

- **Q1【已定】** 秒杀商品池 = 所有后台可售商品（会员卡/权益卡/私教/团课/商城）→ 多态模型。
- **Q1a【已定】** 划线原价 = 各域**对外售价**（私教 `sale_price`、会员卡 `cardPrice`、权益卡 `currentPrice`），**秒杀价即成交价**；docx"团购价"统一按对外售价理解，不加"团购价"字段。售罄后"恢复团购价售卖"= 恢复该商品原对外售价售卖。
- **Q1b【已定】** P0 **不**为参与域新增服务端价格重算防篡改；秒杀价由服务端按活动写入订单、前端传一致值。**接受**会员卡"信任前端 paySum"等既有缺陷的薅羊毛风险，留待 P2 统一整改。
- **Q1c【建议采纳】** `mk_flash_sale_activity` 加 `biz_type` 做多态（单表），P0 只用到私教/会员卡/权益卡三种取值。
- **Q2【已定】** P0 只接 3 个域（私教/权益卡/会员卡普通卡）的 `createOrder` 分发，会员卡自动续费/团课/商城押后。
- **Q3【会员卡·自动续费】** 委托代扣自动续费卡：秒杀价**只作用首期**（后续 `AutoPayTask` 按 `nextPrice` 原价续费），还是续费也覆盖？（决定自动续费卡是否允许参与秒杀）
- **Q4【会员卡·门槛】** `cardNature=1` 权益性质会员卡（须先持 VIP 权益卡、首页不展示）是否可入秒杀池？抢购时 `ERROR_FIT_CARD_NEED_BENEFIT` 门槛是否保留？
- **Q5【权益卡·动态价】** 秒杀期间是否**暂停**权益卡按购买人数涨价（`currentPrice`），直接用秒杀价？"一人一卡"硬限购与秒杀"限购件数"如何叠加（权益卡实际最多 1 件）？
- **Q6【团课】** 秒杀绑"具体某场次"还是新建"可复用团课商品"概念？"必须持卡会员"前置是否为秒杀拉新放开？团课线上三表 0 行、疑似休眠功能，**本期是否确认上线团课秒杀**（否则可暂缓）？
- **Q7【商城】** 商城会员端 UI 已不存在（2018 遗留），秒杀绑商城商品需先**重建会员端商城页**+补服务端价格权威+处理实物发货——工作量最大，**是否本期做还是单独立项**？
- **Q8【已定】** P0 只做**单次生效 + 总量库存**；循环生效（生效日+每日多时段+每日投放库存）、倒计时预热、售罄两种策略放 P1。
- **Q9【已定】** 限购**叠加**：商品原生限购（私教商品级 `purchase_limit`、权益卡"一人一卡"）与秒杀"每人限购"两道都拦（沿用私教现有的"商品级+活动级双重"范式）。

## 7. P0 落地清单（完成态 + 要动的文件）
