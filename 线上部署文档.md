## 0. 一句话速查（TL;DR）

- **后端**：`Backend/` → JDK8 下 `mvn clean package` 出 `target/hzjsf.war` → 解压覆盖到 `/www/wwwroot/shilijsf/`（**务必保住 `WEB-INF/classes` 下的线上 `*.properties`，别被 war 内的开发配置覆盖**）→ 在宝塔「Java 项目一键部署」里**重启** shilijsf 对应的 Tomcat 实例。
- **前端(admin)**：`admin/` → Node16 下 `npm run build` 出 `dist/<版本号>/` + `dist/config/index.js` → **先把 `config/index.js` 里的 `baseUrl` 改成 `https://shilijsf.shilisports.com`**（构建默认是 renren 的 demo 域名，是个坑）→ 把 `<版本号>/` 目录和 `config/index.js` 上传到 `/www/wwwroot/shilijsf/admin/`。前端是纯静态，**不用重启、不用重载 Nginx**。

## 2. 实测架构总览

```
浏览器
  │  https://shilijsf.shilisports.com
  ▼
Nginx (:80 / :443)            root = /www/wwwroot/shilijsf
  ├─ location ~ /admin(.*)        → 直接读静态文件（admin 前端 SPA）
  ├─ location ~ \/$               → 302 跳转到 /admin
  └─ location /（其余全部）        → proxy_pass http://127.0.0.1:8082   ← 含 /api /sys /statics /h5 /coupon
                                     （带 WebSocket upgrade 透传）
       │
       ▼
Tomcat 9 实例（宝塔「Java 项目一键部署」的多实例模式，以 www 用户 jsvc 守护运行）
  · catalina.base / catalina.home = /www/server/web_site/shilijsf.shilisports.com
  · HTTP Connector 端口 = 8082      · shutdown 端口 = 8243
  · JDK = /usr/java/jdk1.8.0_121（Java 8）
  · <Host name="shilijsf.shilisports.com">
      <Context docBase="/www/wwwroot/shilijsf" path="" reloadable="true"/>   ← ROOT 应用，path 为空
       │
       ▼
/www/wwwroot/shilijsf/           ← 站点根 = 解压后的后端 WAR + 独立部署的 admin 前端
```

**要点（都是实测）**：
1. 后端**不是**丢进 `tomcat/webapps` 自动部署的 war，而是把**解压后的 war 内容直接当作站点根 `/www/wwwroot/shilijsf`**，由该 Tomcat 实例通过 `docBase` 挂成 `path=""` 的 ROOT 应用。所以线上接口路径**没有** `/renren-security` 前缀（本地开发才有），是 `https://shilijsf.shilisports.com/api/...`、`/sys/...`。
2. Tomcat 端口是 **8082**（不是本地的 8080）。
3. `admin/` 前端是**纯静态**，由 Nginx 直接吐，不经过 Tomcat。
4. 同机还跑着另一个站点 `hd.shilisports.com`（独立的 Tomcat9 实例，进程另算），**部署时别搞混实例**。

## 3. 站点根目录职责（实测 `/www/wwwroot/shilijsf/`）

| 路径 | 归属 | 部署时 |
|---|---|---|
| `WEB-INF/` | 后端（web.xml、classes、lib、views） | 随后端 war 更新 |
| `WEB-INF/classes/*.properties` | **线上配置**（db/config/comm 等） | ⚠️ **必须保留，勿被 war 内开发配置覆盖** |
| `META-INF/` | 后端 war 元信息 | 随后端 war 更新 |
| `statics/` | 后端自带静态资源 | 随后端 war 更新 |
| `h5/` `coupon/` | 业务 H5/优惠券静态页 | 随后端 war 更新 |
| `admin/` | **前端 admin（独立部署）** | 随前端更新，**后端 war 里没有此目录，不会被后端部署动到** |
| `index.html` `404.html` `favicon.ico` `address.json` | 根静态资源 | 视 war 内容而定 |
| `.well-known/` `*.txt`（zvcgluu…、5a4e0…） | SSL/域名校验文件 | **勿删**（Nginx 有专门放行规则） |

## 4. 后端部署（Backend → hzjsf.war）

### 4.1 构建（本地）
```bash
cd Backend
# 必须 JDK8（本机默认 JDK 会变，编译前确认 JAVA_HOME 指向 Corretto8 / JDK1.8）
mvn clean package        # surefire 配了 skipTests=true，默认跳测试
# 产物：Backend/target/hzjsf.war
```
> 首次编译若报 `sun.dc.pr.PRError` 之类，是 `TeamClassOrderController.java` / `IncomePayDetailServiceImpl.java` 里的无效 import，删掉即可。

### 4.2 上传并覆盖到站点根（⚠️ 保配置）

**关键**：`hzjsf.war` 是用**开发配置**打出来的（`WEB-INF/classes` 下的 `db.properties`/`config.properties`/`comm.properties` 指向开发库/开发 URL）。线上这几个文件是**生产配置**，解压覆盖时**绝不能被覆盖**。

> 实测佐证：线上 `db.properties`、`config.properties` 的修改时间是 `Jun 4`（部署时配好后就没再动），而 `comm.properties`、`generator.properties` 是 `Jul 9`（跟着上次 war 一起被更新了）。说明历史部署**只保护了 db/config，comm 曾被 war 覆盖过**——请重点核对 `comm.properties`（项目 URL、融云、百度人脸、二维码路径）当前是否为生产值。

**路径 A · 宝塔面板（推荐，贴合现有习惯）**：
1. 宝塔 → 软件商店，确认 Nginx / MySQL / Redis / **Java 项目一键部署** 均运行中。
2. Java 项目一键部署 → 找到 shilijsf 项目（映射 Tomcat9，项目目录 `/www/wwwroot/shilijsf`）。
3. 按面板指引上传/更新 war（面板会解压到项目目录）。
4. **上传后立即核对** `WEB-INF/classes` 下 `db.properties`/`config.properties`/`comm.properties` 是否仍为生产配置，被覆盖了就改回来。
5. 面板里**重启**该 Java 项目。

**路径 B · 命令行（理解原理 / 面板不可用时备选，实测环境支持）**：
```bash
# 服务器上，务必先备份整个站点根，再动
cd /www/wwwroot
cp -a shilijsf shilijsf.bak.$(date +%Y%m%d%H%M)         # 备份，用于回滚

# 先备份线上生产配置（三件套）
mkdir -p /tmp/shilijsf_prop_bak
cp -a shilijsf/WEB-INF/classes/{db,config,comm}.properties /tmp/shilijsf_prop_bak/

# 解压新 war 到临时目录，再覆盖业务文件（这里用整体覆盖示意；生产建议精确覆盖 WEB-INF/statics/h5/coupon 等）
mkdir -p /tmp/hzjsf_new && cd /tmp/hzjsf_new && unzip -oq /path/to/hzjsf.war
cp -a /tmp/hzjsf_new/. /www/wwwroot/shilijsf/

# 用备份把生产配置盖回来（关键一步）
cp -a /tmp/shilijsf_prop_bak/*.properties /www/wwwroot/shilijsf/WEB-INF/classes/

# 修正属主（站点以 www 用户运行）
chown -R www:www /www/wwwroot/shilijsf

# 重启 Tomcat 实例：进程是 jsvc/commons-daemon 守护，优先用宝塔面板「重启」按钮；
# 命令行重启方式请以宝塔该实例 bin/ 下脚本为准，勿凭空 kill。
```
> ⚠️ 路径 B 的解压/覆盖命令为**参考骨架**，首次执行前请再核对：war 内是否含会误覆盖 `admin/`、`.well-known/`、校验 txt 的内容（正常不含）；以及是否需要精确到只覆盖 `WEB-INF`/`META-INF`/`statics`/`h5`/`coupon`。

### 4.3 只改后端配置（不换 war）
直接改 `/www/wwwroot/shilijsf/WEB-INF/classes/` 下的 `db.properties` / `config.properties` / `comm.properties`，然后**重启该 Tomcat 实例**生效。

### 4.4 后端日志
`/www/server/web_site/shilijsf.shilisports.com/logs/catalina-daemon.out`（实测该实例的 stdout/stderr 输出到此）。

## 5. 前端部署（admin → 静态产物）

### 5.1 构建（本地）
```bash
cd admin
nvm use                  # 必须 Node 16（webpack3 在新 Node 下会挂）。本机 nvm 若被 Homebrew 压住，需手动把 v16 的 bin 前置到 PATH
npm run build            # 实为 gulp：先 webpack 打包，再生成带时间戳的版本目录
```

**产物结构（实测 `admin/dist/`）**：
```
dist/
├─ <版本号>/static/...     # 版本号 = 年月日时分，如 2607091041（gulp 用 new Date() 生成）
├─ config/index.js         # 由 static/config/index-prod.js + init.js 合并，version 字段被替换为本次版本号
└─ index.html              # 极简入口：只 document.write 加载 ./config/index.js（一般不变）
```
版本号机制（renren-fast-vue 的 CDN 版本号方案）：`config/index.js` 里 `version='2607091041'` → `cdnUrl='./2607091041'` → 页面从 `./2607091041/static/...` 加载 css/js。**换版本 = 上传新时间戳目录 + 把 config 的 version 指过去**，天然支持灰度与回滚。

### 5.2 ⚠️ 坑：构建产物的 baseUrl 是 demo 域名，必须改

源码模板 `admin/static/config/index-prod.js` 里 `baseUrl` 是 renren 官方 demo：
```js
window.SITE_CONFIG['baseUrl'] = 'http://demo.open.renren.io/renren-fast-server';   // ← 错误默认值
```
构建出的 `dist/config/index.js` 会原样带上这个 demo 域名（实测本地 `dist/config/index.js` 确实是 demo 域名，而线上 `admin/config/index.js` 是 `https://shilijsf.shilisports.com`）。

**因此每次部署前端前，二选一**：
- **构建前**：把 `admin/static/config/index-prod.js` 的 `baseUrl` 改成 `https://shilijsf.shilisports.com`（改完可考虑提交回仓库，一劳永逸）；或
- **上传后**：把服务器 `/www/wwwroot/shilijsf/admin/config/index.js` 的 `baseUrl` 手动改成 `https://shilijsf.shilisports.com`。

> 说明：业务接口的实际请求前缀走的是 `.env.production` 的 `VUE_APP_URL=https://shilijsf.shilisports.com`（已正确、已提交），所以即便漏改 `SITE_CONFIG.baseUrl`，主流程接口未必立刻报错；但线上历史版本 `baseUrl` 就是生产域名，为避免个别引用出错，**按线上现状统一改成生产域名**。

### 5.3 上传
把本次构建的 **`dist/<版本号>/` 整个目录** 和 **`dist/config/index.js`**（baseUrl/version 已确认）上传到：
```
/www/wwwroot/shilijsf/admin/
├─ <版本号>/          # 新增本次时间戳目录，如 2607091041/
├─ config/index.js    # 覆盖：version 指向新时间戳目录，baseUrl=生产域名
└─ index.html         # 一般不变；有改动才覆盖
```
- 前端是纯静态，**上传即生效**（clear 浏览器缓存即可），**不用重启 Tomcat、不用 reload Nginx**。
- 上传后确认属主 `www:www`（宝塔文件管理上传通常自动处理；命令行上传后 `chown -R www:www /www/wwwroot/shilijsf/admin`）。

### 5.4 前端回滚
旧时间戳目录不删（如线上现存 `2607091041/`，本地 dist 还留着 `2607062308/`）。回滚只需把 `config/index.js` 的 `version` 改回上一个时间戳目录名即可，**秒级回滚、无需重新构建**。

## 9. 故障排查（实测端口/路径版）

| 现象 | 排查方向 |
|---|---|
| `/admin` 空白 | `admin/` 目录文件是否完整；`config/index.js` 的 `version` 是否指向**已上传**的时间戳目录；清浏览器缓存 |
| 接口 404 / 登录后功能报错 | `config/index.js` 的 `baseUrl` 或 `.env.production` 的 `VUE_APP_URL` 是否为生产域名；接口路径是否误带 `/renren-security`（线上不该有） |
| **502 Bad Gateway** | Tomcat 实例是否在跑、**8082** 是否在监听（`ss -tlnp | grep 8082`）；`catalina-daemon.out` 是否启动报错 |
| 连不上库 / 登录失败 | `WEB-INF/classes/db.properties`、`config.properties` 是否为生产配置（部署时被 war 覆盖成开发配置是高发原因）；MySQL/Redis 服务状态 |
| 改了 `api` 的 Mapper XML 不生效 | `api` 模块 Mapper XML 需**重启** Tomcat 才生效（`sys` 的才有热刷新） |

## 10. 关键坑汇总（部署前过一遍）

1. 🔴 **后端 war 会带开发配置**：解压覆盖时保住 `WEB-INF/classes` 下 `db/config/comm.properties` 的生产值（尤其 `comm.properties` 历史上被覆盖过）。
2. 🔴 **前端 `config/index.js` 的 `baseUrl` 默认是 renren demo 域名**：每次部署改成 `https://shilijsf.shilisports.com`。
3. 🟠 **前端换版本必须换时间戳目录**：js/css 有 12h 缓存，原地覆盖同名文件会因缓存不生效。
4. 🟠 **Tomcat 端口是 8082、上下文是 ROOT（无 /renren-security 前缀）**：和本地开发（8080 + /renren-security）不同。
5. 🟠 **同机有 `hd.shilisports.com` 另一套 Tomcat 实例**：重启/改配置别选错实例。
6. 🟠 **JDK 必须 8**（线上 `jdk1.8.0_121`；本地编译也要 JDK8）。
