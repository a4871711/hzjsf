# 失历健身私教管理后台 — 开发任务拆分（逐步实施清单）

> 依据：《失历健身私教管理后台_详细实现文档.md》（总则第 0 章定义全局约定/复用判定/表总览/菜单权限/订单号后缀，第 1~6 章为六个业务域）。需求 DDL 以《私教需求文档》第 21 节为唯一事实来源。命名/返回/分页/权限/事务/Mapper 热刷以总则 §0.2 为准。
> 用途：把六域实现拆成 **24 个可独立交付、可单独验收的步骤**，按「地质分层」推进——地基先就位，主数据次之，核心交易居中（最难、依赖最多），营销/资金随后，运营/报表收尾，前端最后联调。每步给「目标 / 新建文件 / 改动文件 / 关键点 / 验收」，并标注依赖步号、里程碑与可并行项。
> 体例对齐同仓库 `Backend/需求文档/VIP会员权益转让_开发任务拆分.md`。
> 路径约定：后端根 `Backend/src/main/java/com/dlc/`，Mapper XML 在 `Backend/src/main/resources/mapper/{api,sys}/`，SQL 在 `Backend/sql/`，前端根 `admin/src/`。

## 风险与回归点（与现有 SysCoach / SysCoupon / SysFitCard / 团课 / 支付 的兼容）

> 私教系统几乎全是新建表与新建类，主要风险在**与既有支付回调链、既有同名实体、既有定时任务**的交叉点。下列为必查回归项。

1. **订单号后缀回调错分（最高危）**：`PayController` 的 `wxNotify`/`rechargeCallBack`/`zfbNotify` 三处用 `orderNo.substring(len-1)` 分发，现有占 `CARD=2/GOODS=3/PRIVATE_CLASS=4/TEAM=5/VIP_CARD=6/VIP_FEE=7`。**新私教订单严禁复用 "4"**（"4" 属旧私教课 `PRIVATE_CLASS_ORDER_TYPE`，复用会让新私教订单回调进旧逻辑）。本拆分用 8/9/a + 一个私教单字母后缀，新增分支必须在**三处** substring 链同步加，且加在 if-else 末尾不破坏既有分支顺序。回归：跑通卡/商城/团课/VIP 既有四类回调不受影响。

2. **`pt_coach` vs 旧 `SysCoachEntity`(coach 表) / `pt_coach_store_rel` vs `SysStoreCoachEntity`**：两套并存、**不复用不迁移**。旧表继续服务「教练入驻审核」与旧团课/旧私教；私教域全部走 `pt_*`。回归：旧教练审核、旧门店教练关联、旧 `StoreCoachEvaluate` 评价功能不受影响；勿在私教页误调旧接口。

3. **`mk_coupon` vs 旧 `SysCouponEntity`(sys_coupon) / `mk_member_coupon` vs `UserCouponEntity`(user_coupon)**：并存不迁移，**状态枚举不同**（新 1上架/0下架，旧 1上架/2下架）。私教链路只认 `mk_*`。回归：旧券（抖音/美团核销、绑 `fit_card`）继续可用；前端勿照抄旧状态值。

4. **`pt_coach_fee_rule` 跨域同名表**：教练域(扩展版)与运营域(基础版)都引用。**第 1 步已仲裁为扩展版只建一次**，运营脚本不建。风险=若两域各自建表会冲突/字段缺失。回归：报表成本命中与教练分成结算用同一张表、同一 `matchFeeRule`，不得出现两套 DDL。

5. **`pt_installment_rule` 跨域**：建表归商品域、CRUD 在商品表单、资金域只读。风险=资金域误建第二张。回归：分期配置改动只在商品表单生效，资金域读到的是同一行。

6. **既有定时任务争锁/扩展**：`UpdateOrderStatusTask`(每小时) 被**扩展**加私教待支付超时分支——勿破坏其卡/商城/私教课/团课既有分支；新增 6 个扫描/过期任务(权益过期/爽约/逾期/续费/异常/团课转私教) cron 必须**错开** `0 0 0`(会员过期)、`0 0 0/1`(订单状态)、`0 0 2`(人脸清理)、`0 0 8`(自动代扣)。回归：既有任务执行时间与逻辑不变。

7. **`IncomePayDetailServiceImpl` 扩展**：新增私教/储值/分期后缀的反查分支（仿 VIP 后缀 6/7 写法），勿改动既有卡/商城/团课/VIP 分支的记账口径。回归：既有四类订单记账金额/门店归属不变。

8. **储值与既有 `SysWalletDetailEntity` 区分**：新建 `pt_member_wallet*` 是私教专用余额主账户，**与旧提现/充值明细 `SysWalletDetailEntity`(type=1/2/3) 不共用**（需求：储值仅用于私教消费）。回归：旧提现/充值明细功能不受影响，勿混账。

9. **门店数据隔离一致性**：所有 sys 列表/详情/操作必须 `getStoreIds()` 过滤（超管 user_id=1 放行）。`AbstractController` 不直接给 storeIds，需按现有范式取登录用户门店字段。回归：与现有 sys 模块隔离行为一致，无越权看他店数据。

10. **api Mapper XML 必重启**：所有 `mapper/api/*.xml`（交易/营销/资金/运营/教练 api/商品 api 共约 20+ 个）与 `PayController`/`ConfigConstant` 改动**必须重启 Tomcat** 才生效（`MybatisMapperRefresh` 只热刷 `mapper/sys/`）。回归排查支付/预约「没生效」时先重启再查。

11. **快照不可回查**：订单/权益/评价/分期计划的课时/时长/有效期/商品名/类型/券/活动/会员名均为下单/写入快照；商品/券/规则改名改价**不回写历史**。回归：改主数据后历史订单展示与金额口径不变。

12. **团课依赖（附赠团课权益核销 + 团课转私教统计）**：附赠权益核销要求团课预约侧接入「优先扣赠送权益」、团课转私教扫描读现有团课出勤/购课表。这两处依赖现有团课模块，若未接入则附赠权益无数据/名单为空（属正常非 bug）。回归：现有团课预约/出勤/订单功能不受影响。
