# 停卡功能重构 —— 需求与实现方案

> 2026-07-08。本文档描述停卡功能从「开放式免费停卡」重构为「免费额度 + 付费停卡规则」的完整需求与技术实现，是本次改动的权威口径。

## 1. 背景与目标

### 1.1 旧版停卡（已废弃）

- 有效权益会员可对生效中的会员卡申请停卡，**不预设天数**，恢复时按实际停卡天数顺延到期日。
- 限制：每张卡每自然月 1 次（DB 唯一键 `uk_card_month` 兜底）+ 全年 12 次，全程免费。
- 痛点：无法区分免费/付费权益，停卡时长不可控，无收费能力。

### 1.2 新版目标

| 维度 | 新规则 |
|---|---|
| 停卡对象 | 只能停「**权益卡性质**」的会员卡（`fit_card.cardNature=1` 对应的 `card_order`），卡须属本人、`status=4` 生效中、未过期 |
| 前提 | 有效权益会员（名下有正常且未过期的 `vip_benefit`） |
| 免费停卡 | **按用户**滚动 30 天 1 次；每次自选天数 **1~7 天** |
| 付费停卡 | 免费额度用完后，按该会员卡对应权益卡商品绑定的「停卡规则」选档付费（微信支付），回调成功后停卡生效；权益卡未绑定规则 = 无付费停卡 |
| 生效方式 | **申请时预顺延**：生效当刻把 `card_order.validityDate` 顺延 N 天并记 `end_time = start_time + N`，到期零动作（不依赖定时任务） |
| 提前取消 | 停卡中可手动取消：未使用天数从有效期扣回（已停不足 1 天按 1 天算），**付费不退款** |
| 旧限制 | 每自然月 1 次、全年 12 次 **全部废弃**（删 `uk_card_month`、`pause_month`） |

## 2. 业务规则细则

### 2.1 免费额度（滚动 30 天，按用户）

- 判定：取该用户最近一条 `pause_type=0` 且 `status IN (0,1,2)` 的记录 `start_time`，`start_time + 30天 <= now` 才有额度。
- **已取消的免费停卡仍占额度**（防"停 1 天就取消 → 无限白嫖"）。
- 额度按用户而非按卡：一个权益会员名下多张卡共享同一免费额度；申请免费停卡时对 `user_info` 行加 `FOR UPDATE` 锁，防两张卡并发各占一次。

### 2.2 停卡规则（付费档位）

- 新表 `vip_pause_rule`：一条规则 = 名称 + 多档阶梯 `tiers_json`（`[{"days":3,"price":100},{"days":10,"price":200}]`）+ 启用状态。
- 权益卡商品 `vip_benefit_card.pause_rule_id` 单绑定一条规则（同「转让费用规则」`fee_rule_id` 模式）。
- 规则来源：用户**最新一张有效** `vip_benefit` → 其 `vip_card_id` → `vip_benefit_card.pause_rule_id`。
- 兜底：规则不存在 / `status=0` 停用 / 档位为空 → 视为「未开通付费停卡」（同 `calcTransferFee` 的免费兜底思路，但停卡是拒绝而非免费）。
- 校验：后台保存规则时强校验 tiers（非空 JSON 数组、1~10 档、days 为 1~365 正整数不重复、price>0）；申请付费停卡时**金额与天数一律以后端规则为准，不信前端**。

### 2.3 状态机（`card_pause_record.status`）

```
付费: 10 待支付 ──支付回调──▶ 0 生效 ──到期──▶ (查询时显示"已结束" displayStatus=99, 不写库)
                │                    │
                │用户对同卡再次申请    │用户提前取消
                ▼                    ▼
              3 已关闭             2 已取消(扣回未用天数)
免费: 直接插入 0 生效
1 已恢复 = 旧版历史数据专用，新代码不再写入
```

- 到期状态**查询时惰性计算**：`status=0 AND end_time <= NOW()` 显示为 99「已结束」，库里不更新（预顺延模式下到期无需任何动作）。
- 旧开放式存量记录（`end_time IS NULL` 且 `status=0`）：取消时走 legacy 分支，沿用旧 resume 语义（按实际天数**顺延**，status→1）。

### 2.4 提前取消

- 守卫：`status=0 AND end_time > NOW()`（SQL 层幂等闸，命中 0 行直接返回）。
- `usedDays = max(1, ceil((now - start_time) / 1天))`；`refundDays = pause_days - usedDays`；`refundDays > 0` 时 `validityDate` 扣回（`DATE_SUB`）。
- 付费停卡取消**不退款**，仅扣回未用天数。

### 2.5 未支付单收口

- 同卡再次 apply 时先把该卡存量 `status=10` 记录关闭为 3（顶替），再判「该卡是否已有生效中/待支付停卡」。
- 支付回调对 `status IN (10,3)` 均可激活为 0（覆盖"用户关单后微信仍付款成功"的竞态）；对已生效/已取消的单幂等返回。
- 不加定时任务清理待支付单。

## 3. 数据库设计

变更脚本：`Backend/sql/card_pause_v2.sql`（**必须先于新代码部署**，mapper 为手写显式列清单，缺列直接报 Unknown column）。已于 2026-07-08 在开发库 `shilijsf` 执行。

1. **新表 `vip_pause_rule`**：`pause_rule_id` PK、`rule_name`、`tiers_json VARCHAR(2048)`、`status TINYINT DEFAULT 1`、`created_date`（结构照 `vip_fee_rule`）。
2. **`vip_benefit_card` 加列** `pause_rule_id BIGINT NULL`（AFTER `fee_rule_id`），NULL=不关联。
3. **`card_pause_record` 升级**：
   - DROP：唯一键 `uk_card_month`、列 `pause_month`。
   - ADD：`pause_type`(0免费/1付费)、`pause_rule_id`、`amount DECIMAL(10,2)`、`pay_order_no VARCHAR(40)`(UNIQUE `uk_pause_pay_order`)、`transaction_id`、`pay_time`、`cancel_time`、`actual_days`；KEY `idx_user_type_start(user_id, pause_type, start_time)`。
   - 语义变更：`pause_days`=计划停卡天数（申请时写入）；`end_time`=计划结束时间；`status` 见 2.3 状态机。
4. **菜单**：「停卡规则」插入「VIP权益」目录（`url='modules/sys/vipPauseRule.html'`，perms `sys:vipPauseRule:list/info/save/update/delete`，NOT EXISTS 防重）。超管自动可见；非超管角色需在 `sys_role_menu` 另行授权。

## 4. 接口设计

### 4.1 移动端 api（`/api/cardPause/*`，token 鉴权走 BaseController）

| 接口 | 参数 | 返回 / 说明 |
|---|---|---|
| POST `/precheck` | — | `{freeAvailable, nextFreeDate, maxFreeDays:7, tiers:[{index,days,price}]}`；tiers 空=未开通付费停卡 |
| POST `/apply` | `cardOrderId, pauseType(0/1), pauseDays(免费1~7), tierIndex(付费档下标)` | 免费：`{pauseId, needPay:false}`（立即生效）；付费：`{pauseId, needPay:true, orderNo, paySum}`（orderNo 交 `/wx/proPay` 拉起支付） |
| POST `/cancel` | `pauseId` | 提前取消（新） |
| POST `/resume` | `pauseId` | 兼容旧客户端，委托 cancel |
| POST `/list` | 分页 | 记录含新字段 + `displayStatus` |

**错误码**（`CodeAndMsg`，占用 -75~-79；-66/-67 删除）：

| 码 | 含义 |
|---|---|
| -71 | 开通权益会员后才可停卡（保留） |
| -68 | 停卡状态异常（卡不合法 / 已在停卡中 / 取消守卫不通过，保留） |
| -75 | 30天内已使用过免费停卡 |
| -76 | 免费停卡最长7天 |
| -77 | 该卡未开通付费停卡 |
| -78 | 停卡档位不存在 |
| -79 | 仅权益卡性质的会员卡可停卡 |

### 4.2 后台 sys（`/sys/vipPauseRule/*`，照 `vipFeeRule` 五件套）

`list / info/{id} / save / update / delete`；save/update 前 `validateTiersJson` 强校验（复用 -70 格式错误码）；delete 前 `countCardRefByRuleId > 0` 拒删。停卡记录 `/sys/cardPause/list` 适配新字段与 `displayStatus`，删除 `yearCount`。

### 4.3 支付链路（复用现有微信支付，不建新订单表）

- 支付单号：`OrderNoGenerator.getOrderIdByTime() + "c"`（新常量 `ConfigConstant.CARD_PAUSE_FEE_TYPE="c"`；2~9/a/b 已被占用），存 `card_pause_record.pay_order_no`。
- 拉起支付：uniapp 复用 `/api/wx/proPay` + `uni.requestPayment`（照转让服务费页写法）。
- 回调路由：`PayController.wxNotify` 与 `WxPayController.proPayNotify` 均按单号后缀分发，各加后缀 `c` 分支 → `cardPauseService.payCallback(orderNo, money, transactionId, payType)`；`wxNotify` 的「跳过重复记账」后缀清单同步加 `c`，收入记账（`incomePayDetailService.saveIncomePayDetail`）在 service 单事务内完成（同转让费后缀 `7` 模式）。
- 幂等：回调 `FOR UPDATE` 查单，已生效/已取消直接返回；`activateByPayOrderNo` 带 `status IN (10,3)` 条件，命中 0 行不重复顺延。
- 金额校验：`payCallback` 复查实付金额，不等于建单锁定的档位价则 `return 0`（不生效、不顺延、不记账），照 `VipTransferServiceImpl.payFeeCallback` 既有防线——`proPay` 的 `paySum` 取自前端不可信。
- 加锁顺序：`payCallback` 与 `cancel` 均按「先锁 `card_order` 行、再锁停卡单」加锁（与 `apply` 同序），避免同卡并发的 AB-BA 死锁。

时序：`apply(付费)` 建 status=10 单 → 前端 `/wx/proPay` → `uni.requestPayment` → 微信回调 `proPayNotify`/`wxNotify` → `payCallback`：激活 10→0、写 start/end/pay_time、顺延 validityDate、记账 → 前端延时刷新看到「停卡中」。

## 6. 部署顺序与验证清单

1. 执行 `Backend/sql/card_pause_v2.sql`（先于代码）。
2. 部署后端 WAR 并重启（api mapper XML 改动必须重启）。
3. 非超管角色在「角色管理」为新菜单授权。
4. 部署 admin 前端；uniapp 由 HBuilderX 发版。

验证要点：

- [ ] admin 建停卡规则（3天100 / 10天200）→ 权益卡关联 → 删被引用规则被拒。
- [ ] `/precheck`：额度与档位正确；非权益会员 -71。
- [ ] 免费 apply 8 天 → -76；成功后 `validityDate` 顺延且 `end_time=start+N`；30 天内二次免费 → -75；非 `cardNature=1` 的卡 → -79。
- [ ] 付费 apply 返回后缀 `c` 单号；回调后 10→0、顺延、`sys_income_pay_detail` 恰一条；重放回调不重复顺延。
- [ ] 取消：`validityDate` 扣回 `pause_days - max(1,已停天数)`；到期后取消 → -68。
- [ ] 旧开放式存量记录（end_time NULL）取消走 legacy 顺延。

## 7. 与既有功能的关系

- 模板对应：`vip_pause_rule` ↔ `vip_fee_rule`；`pause_rule_id` ↔ `fee_rule_id`；`vipPauseRule.vue` ↔ `vipFeeRule.vue`；支付后缀 `c` ↔ 转让费后缀 `7`。
- 依赖工作区同批改动：`fit_card.cardNature`（卡种性质）——停卡对象校验依赖它；部署时 `vip_benefit_bind_fit_card.sql` 须一并执行。
- 权益卡商品管理（vipCard.vue / SysVipCardController）只是多一个绑定字段，不影响购买/转让流程。
