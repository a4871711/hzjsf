# 连续包月卡「随单开通权益会员」详细设计（方案B）

## 背景与目标
购卡续费页原有「随单开通权益会员」入口只对**非连续卡（autoPay=0）**开放（前后端双闸门：`card_renewal.vue` 的 `showVipBuy` + `CardOredrController` 的 `autoPay==0` 判断）。原因是连续卡走的是**微信委托代扣签约**（跳签约小程序 wxbd687630cd02ce1d），不是一次性支付，一次性权益卡的钱没有落点。

本需求让**连续包月卡也支持随单开通权益会员**，采用**方案B**：委托代扣签约的**首期扣款金额 = 会员卡权益价 + 权益卡实时价**，后续每期恢复月费；并改造每日自动续费定时任务，对**已激活**权益会员按会员卡权益价 `benefitPrice` 扣款。

## 已确认的决策
1. **微信模板前提**：方案B靠首期加价实现，前提是 `plan_id` 模板支持「不定额/按需扣款」且单次上限 ≥ 首期最大金额。**开工前由运营核实**（现有阶梯续费能按 nextPrice 扣不同金额，已反证模板大概率支持不定额；核实重点在单次上限）。
2. **权益卡过期后档位**：权益会员期间续费次数 `buyCount` 照常累加；权益卡 365 天过期后，续费**按当前累计档位**（通常已是最高档 nextPrice3）接续，不重置。
3. **续费价口径**：已激活权益会员续费，每期按 `fit_card.benefitPrice`（会员卡的权益会员专享价，与购卡页展示权益价同源）扣。

## 关键现状（含代码位置，均已核验）
- 签约参数不含金额：`WxPayServiceImpl.appPreenTrustWebParams`（:107-132）纯授权模板，`plan_id` 写死（:84-88）。
- 每期扣款金额动态传入：首期 `firstPapPayment` 用 `realPayment`（:193）；续费 `createAutoPayCardOrder` 用阶梯 `price`（:266-286→:309）。两处独立。
- 首期代扣成功回调 `WxPayController.papPaymentNotify`（:180-238）只 `updateCardOrder` 开卡，**缺权益卡激活**。
- 加购占位与激活现成：`CardOrderServiceImpl`（:219-238）建 `vip_benefit` status=9 占位；`VipBenefitServiceImpl.activateAttached`（:118-136）幂等激活（按 `source_order_no`+status=9）。
- 权益会员判定现成：`vipBenefitMapper.countValidByUser`（`VipBenefitMapper.xml:137-141`：`status=0 且 expire_time 空或未过期`）。
- 权益卡状态机（`vip_benefit.status`）：9待支付 / 0正常 / 1已转出失效 / 2已冻结 / 3已过期。过期/转让/停卡后 `countValidByUser` 归零 → 续费**自动降级**，零额外代码。

## 关键流程（连续卡 + 加购）
```
选连续卡+勾权益卡 → createOrder(带buyVipCardId)
  → 后端建续费单 realPayment=权益价+权益卡价 + vip_benefit status=9 占位
  → getWxpayData 签约参数 → 跳微信签约小程序 → 签约成功 preenTrustNotify
  → firstPapPayment 发首期代扣(realPayment)
       成功 → papPaymentNotify → updateCardOrder 开卡 + activateAttached 激活权益卡(9→0)
       失败 → 取消订单(status=5)+微信解约 + cancelAttached 清占位
之后每月8点 AutoPayTask → createAutoPayCardOrder
  → countValidByUser>0(权益卡未过期)? 是→扣 benefitPrice / 否→扣阶梯 nextPrice
```

## 边界与错误处理
| 场景 | 处理 |
|---|---|
| 首期代扣失败 | 取消订单 + 微信解约 + 清 status=9 占位（D） |
| 续费回调丢失 | SQL NOT IN 近2天兜底（现有 bf43fd7），次日不重复选中 |
| 权益卡过期/转让/停卡 | `countValidByUser` 归零 → 续费自动回阶梯价（天然） |
| 重复回调 | `activate`/`updateCardOrder` 均幂等（命中0行/status 守卫），不重复记账 |
| 首期当刻权益卡未激活 | 首期金额签约时按「权益价+权益卡价」定，与「续费按身份判定」是两套独立逻辑，不冲突 |

## 实现记录与对抗审查修正
最终实际改动 **5 个文件**：`CardOredrController.java`(T-A)、`WxPayController.java`(T-B)、`WxPayServiceImpl.java`(T-C)、`card_renewal.vue` + `package-details.vue`(T-E)。`VipBenefitMapper.java/.xml` 净零改动。JDK8 `mvn clean compile` → **BUILD SUCCESS**。

3 个独立视角对抗审查发现并**已修复**的必修问题：
- **[高] T-C 曾是死代码**：原用 `fitCardMapper.selectByPrimaryKey()`，但 api 该 SQL **未 SELECT benefitPrice 列**，`getBenefitPrice()` 恒 null → 续费权益价永不生效。**已改用 `getFitCardInfo()`**（`SELECT f.*`，含 benefitPrice）。
- **[中] 0 元首单权益卡永不激活**：`firstPapPayment` 的 `realPayment==0` 分支只开卡、无激活。**已补 `activateAttached(orderNo)`**。
- **[中] 首期失败物理删占位不可补救**：原 `cancelAttached` 物理 DELETE status=9 占位，但"同步失败≠未扣款"（SYSTEMERROR 等），异步若实扣则权益卡永久丢失。**已撤销**，保留 status=9 占位（不被 countValidByUser 认可、无副作用，晚到回调可补激活）。
