# VIP 会员权益转让 — 需求梳理与实现方案

> 来源规则文档：《健身房VIP会员权益转让管理规则.docx》
> 目标系统：健身房管理系统 hzjsf。仓库现为前后端一体：`Backend/`（Java 后端，Spring MVC + MyBatis-Plus + Shiro，后台 `sys` + 移动端 `api` 两套接口）、`admin/`（后台管理前端，Vue 项目）。后端接口写在 `Backend/`，后台管理页面写在 `admin/`。
> 本文用途：把业务规则逐条翻译成可落地的需求，对照现有代码标注「已有 / 需改造 / 新建」，给出数据模型、接口、流程与决策点。
> 状态：关键决策点已确认（第 4 节）；下一步出《详细技术设计》后进入开发。

## 1. 一句话结论

规则文档要新增的，是围绕一张全新的**"VIP 权益卡"**（可单独购买、类似京东年卡 / PLUS 会员的独立权益卡，与普通健身会员卡**平行、相互独立**）的能力：会员先**买下权益卡**，之后可把它**整体过户给另一个人**——发起转让 → **转让人缴服务费** → 后台审核 → **受让人确认接收（不付费）** → 转让生效；受让人**继承转让人剩余的有效期**（到期日不变、不重置，D-10）。

这套能力在你现在的代码里**几乎是全新的**：权益卡这个"可购买的独立产品"系统里压根没有；唯一沾边的影子功能 `transferCard`（后台管理员直接转卡）也没有"会员自助发起 / 服务费 / 审核 / 受让人确认 / 退费"这些环节，只能当参考、不能直接复用。

此外文档还附带几个**独立的小功能**：停卡、实时购买人数显示、首发权益动态涨价、后台拉黑限制。它们和"转让"是并列关系，可拆开单独做。

| **月卡停卡 12 次** | **完全没有停卡功能**（无停卡/冻结/暂停任何代码） | 全新：停卡申请、次数限制、有效期顺延 |

## 4. 关键决策点

### 4.1 已确认（2026-06-26 拍板）

| 编号 | 决策 | 结论 |
|---|---|---|
| **D-1** | "VIP 权益"建模 | **✅ 独立可购买的「权益卡」**（类京东年卡/PLUS 会员）：新建权益卡商品 `vip_benefit_card` + 用户持有实例 `vip_benefit`，**可单独购买**，与会员卡（`fit_card`/`card_order`）平行、相互独立。权益卡本身就是被转让的对象；会员卡仍归原办卡人、负责入场，永不转让。 |
| **D-4** | 转让费用规则（源文档总则5「首次免费/第2次起收/可自定义」与流程(一)「固定原价 10%」两条原文**均已去掉**，改用下面这套） | **✅ 按转让次数分档配置（后台可配的费用表）**：为该权益的**第 1 次、第 2 次及以后、第 3 次及以后……**分别设定费用（**定额**），第 1 次可设为免费；档位可继续往下加，最后一档兜底所有更高次数。收费时按该权益的 `transfer_count` 命中对应档位。费用全部由转让人承担（D-3）。 |
| **D-3** | 谁缴费（源文档(四)1说"被转让人补缴费用"） | **✅ 费用全部由转让人承担、受让人不缴费**（按你的修正，源文档"被转让人补缴费用"作废）：整个转让只有转让人发起时缴的那笔服务费（D-4），**受让人确认接收即可、无需支付任何费用**，确认成功即生效。 |
| **D-5** | 停卡规则 | **✅ 年卡每自然月可停 1 次、全年共 12 次**；停卡期间**有效期顺延**、停卡时不可入场；**无需后台审核**（会员自助）。 |
| **D-9** | 转让次数冲突（前置2「单份只转一次」vs 总则5「第2次起收费、以此类推」） | **✅ 允许权益链式多次转让**：放弃「单份权益仅限转让一次」（前置条件第 2 条、R-16 的「不可二次转让」一并作废）。权益可 A→B→C→… 连续转让；转让费用按**该权益已被转让的次数**分档收取（具体档位见 D-4）。受让权益仍**不支持退费**。注：原本担心的「链式转让=无限续期」副作用，已被 **D-10 彻底消除**（受让人继承剩余有效期、不重置，时长只会越转越少）。 |
| **D-10** | 转让后受让人拿到的有效期口径（你举的"用了半年再转"的例子） | **✅ 受让人继承剩余有效期**：转让只是把"剩余的整份权益"换个持有人，**到期日不变、不重置**（源文档 R-13 的"重算 1 年"作废）。转让人用掉的时长不会复活；链式转让时长越转越少，顺带堵死 D-9 的无限续期漏洞。 |

## 6. 接口与流程设计

### 6.1 接口清单（移动端 `api` + 后台 `sys`）

移动端（`Backend` 的 `api` 模块，`BaseController.getUserId/getUserVo` 取身份）：
- `GET  /api/vipCard/list`、`GET /api/vipCard/detail` 权益卡商品列表/详情（含实时购买人数 + 动态价，R-07）
- `POST /api/vipCard/buy` 购买权益卡（建权益卡订单→支付→回调建 `vip_benefit` 实例）
- `GET  /api/vipCard/myBenefits` 我持有的权益卡
- `POST /api/vipTransfer/quote` 试算转让费用（按该权益已转次数 `transfer_count` 命中对应费用档）
- `POST /api/vipTransfer/apply` 发起转让（校验前置条件→建转让单→生成转让费用支付单，按 D-4 分档）
- `GET  /api/vipTransfer/myList` 我的转让/受让记录
- `POST /api/vipTransfer/withdraw` 转让人撤回（通过后撤回不退费）
- `POST /api/vipTransfer/confirm` 受让人确认接收（**不付费，确认即生效**）
- `POST /api/vipTransfer/reject` 受让人拒绝
- 权益卡购买、转让服务费支付均复用现有 `PayController` 下单 + `wxNotify` 回调（仅微信支付）
- `POST /api/cardPause/apply`、`GET /api/cardPause/list` 停卡（R-06）

后台（接口在 `Backend` 的 `sys` 模块、Shiro 鉴权；**对应管理页面在 `admin/` Vue 前端新增**）：
- `GET/POST /sys/vipCard/*` 权益卡商品增删改查、上下架、动态定价与服务费规则配置
- `GET  /sys/vipTransfer/list` 转让申请列表（待审核/已通过/已驳回/已生效）
- `POST /sys/vipTransfer/audit` 审核通过/驳回（驳回触发退费 + 通知）
- 服务费规则、动态涨价规则配置项（仿 `SysSetController.memberLevel` 在线配置）

### 6.2 转让全流程 + 状态机

`vip_benefit_transfer.status` 流转：

```
[10 待付服务费]
   └─转让人支付服务费成功→ [20 待门店审核]
        ├─审核驳回→ [31 已驳回] →全额退服务费→结束
        └─审核通过→ [40 待受让人确认]（推送受让人）
             ├─受让人拒绝/超时未确认→ [51/52 已拒绝/已超时] →全额退服务费→结束
             ├─转让人撤回（通过后）→ [60 已撤回] →服务费不退→结束
             └─受让人确认接收（不付费）→ [70 已生效]
                   → 过户事务：
                      · vip_benefit.user_id 改为受让人，transfer_count+1（transferable 保持 1，受让人可再次链式转让）
                      · 原持有人该权益 status=已转出失效（立即丧失）
                      · expire_time 不变（受让人继承剩余有效期，到期日延续原权益，D-10）
                      · 写 income_pay_detail（转让人缴的服务费收入，anotherId 记受让人）
                      · 私教课时/储值/运动档案不动（留转让人）
                      · 推送双方"转让完成"
```

事务边界：过户那步必须单事务（命中 `api.service.impl` 事务切面，REQUIRED）。支付回调要做幂等（防重复过户），沿用订单号去重。

### 6.3 前置校验规则集（发起 + 审核两道关，R-08/17）

拒绝转让的条件（任一命中即拦截/驳回）：权益已过期/不在有效期；账号被封禁（`auditStatus=2`）或在黑名单；存在欠费；已办理退卡；重大违规记录；该权益 `transferable=0`（被冻结/拉黑/违规，**注意：受让所得权益默认仍可转**）；受让人门店不在权益适用门店范围内（受让人 `nowStoreId`，即 `store_addr_id`，∉ 权益来源权益卡 `vip_benefit_card.store_addr_ids` → 抛 `ERROR_VIP_STORE_NOT_APPLICABLE`(-59)「受让人门店不在该权益适用的门店范围内」）。

### 6.4 定时任务（复用现有 `api.schedule` + `spring-mvc.xml` 注册）

- 受让人确认超时扫描 → 自动置"已超时" + 退转让费用（**必须定时任务**：要在双方都没打开 App 时也能自动退款，可挂到现有每小时 `UpdateOrderStatusTask` 节奏）。
- 权益到期（R-03）→ **不用定时任务**，靠 `expire_time` 实时判断；不做到期提醒（已确认）。
- 停卡到期自动恢复 + 有效期顺延（可实时判断，也可定时任务，二选一）。

## 8. 完成态 / 验收清单（做成后长什么样）

- 会员在小程序"我的权益"里能看到自己可转让的 VIP 权益，点"转让"→填受让人→看到服务费试算→支付服务费→提交申请。
- 后台"权益转让审核"列表能看到待审申请，1–2 工作日内点通过/驳回；驳回自动退服务费并通知双方。
- 通过后受让人收到推送，在小程序**确认接收（无需付费）**；确认成功的**那一刻**：转让人这份权益立刻失效，受让人立刻拥有全套权益，**有效期继承转让人剩余时长（到期日不变）**；双方收到"转让完成"通知。
- 受让人**可再次转让**该权益（链式转让，转让费用按该权益已转次数分档，见 D-4）；但受让权益本身不支持退费；入场仍走本人账号。转让人的私教课时/储值/运动档案分文未动。
- 黑名单/欠费/过期/退卡/违规的会员，以及受让人门店不在适用门店范围内的转让，发起或审核时被明确拦截并提示原因。
- 月卡会员可在小程序申请停卡，受 12 次/有效期顺延规则约束。
- 权益购买页公开展示实时购买人数；达到配置人数阈值自动按规则涨价，后台不可手改人数。
- 转让时校验受让人门店在权益适用门店（权益卡 `store_addr_ids`）内，否则拦截。

**不做的事（明确划界）**：不支持权益拆分转让、不支持私下交易撮合、不介入双方私下钱款纠纷、不转移卡所有权与个人资产。

## 9. 工作量与风险提示

- 工作量集中在四块：①权益卡（商品 + 购买）+ 权益层 + 转让状态机（核心、最大）；②停卡（中等、独立）；③动态定价/购买人数（小、独立，可先行）；④`admin/` Vue 后台页面（权益卡管理、转让审核工作台、规则配置）。
- 主要风险：过户事务与支付回调的**幂等与并发**（防重复过户/重复退费）；退费依赖微信退款的可用性与对账；**"权益卡"与"会员卡"两套卡概念并存**，移动端文案/客服话术需明确区分，避免用户混淆。
- D-9 链式多次转让 + D-10 受让人继承剩余有效期：两者结合后**不存在无限续期问题**（时长越转越少）；若把 D-4 的分档费用配成"越转越贵"还能进一步抑制高频倒卖；暂不设最大转让次数上限，如需可后续加。
- 建议落地顺序：先做"权益卡购买（含人数/动态定价）+ 停卡"这些相对独立、可单独验收的项 → 再做转让主流程与审核（适用门店校验并入转让前置校验）→ 最后补 `admin/` 配套页面。
