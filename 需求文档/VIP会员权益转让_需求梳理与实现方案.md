# VIP 会员权益转让 — 需求梳理与实现方案

> 来源规则文档：《健身房VIP会员权益转让管理规则.docx》
> 目标系统：健身房管理系统 hzjsf（Spring MVC + MyBatis-Plus + Shiro，后台 `sys` + 移动端 `api` 两套接口）
> 本文用途：把业务规则逐条翻译成可落地的需求，对照现有代码标注「已有 / 需改造 / 新建」，给出数据模型、接口、流程与决策点。
> 状态：**待你拍板关键决策点后才进入开发**（详见第 4 节）。

---

## 1. 一句话结论

规则文档要新增的，是一整套**"会员把自己付费获得的 VIP 权益（=会员卡使用权）整体过户给另一个人"**的能力：发起转让 → 缴服务费 → 后台审核 → 受让人确认并补缴 → 转让生效、有效期重置为 1 年。

这套能力在你现在的代码里**几乎是全新的**——只有一个最接近的影子功能 `transferCard`（后台管理员直接转卡），但它没有"会员自助发起 / 服务费 / 审核 / 受让人确认 / 退费"这些环节，只能当参考、不能直接复用。

此外文档还附带几个**独立的小功能**：停卡、实时购买人数显示、首发权益动态涨价、后台拉黑限制。它们和"转让"是并列关系，可拆开单独做。

## 2. 现状对照：规则里的概念 → 你系统里的对应物

探测了 9 个子系统后的事实地图（均为代码实证，文件路径相对项目根）：

| 规则里的概念 | 你系统现在有什么 | 缺口 |
|---|---|---|
| **"VIP 权益"** | **没有独立的 VIP 对象**。用户付费拿到、带 1 年有效期、归属某 `userId`、能被禁用的东西 = **会员卡**：`card_order`（订单/持卡记录，`validityDate` 到期日、`userId` 归属）+ `device`（卡设备/有效期/状态）。`fit_card` 是卡商品（`costPrice` 原价、`cardPrice` 售价、`validity` 有效天数、`nextPrice` 续费价）。`user_info.level` 是按运动能量算的会员等级，**不是**付费 VIP。 | 需要新建一个"VIP 权益"层来承载可转让的权益（见第 3、5 节） |
| **会员卡所有权（永不可转）** | `card_order.userId` / `device` 归属 `userId` | 规则要求"卡所有权永远归原办卡人，只转权益"——需在模型上把**所有权**与**使用权益**分离 |
| **权益转让流程** | 仅 `SysUserServiceImpl.transferCard`（`/sys/user/transferCard`，**后台管理员**直接把剩余有效期转给另一会员；记账用 `income_pay_detail.payType=12` 转卡 + `anotherId` 接收人ID） | 无会员自助发起、无服务费、无审核、无受让人确认、无退费、无超时——全部要新建 |
| **服务费 / 收费** | 支付体系成熟：`PayService` 支持微信(2)/支付宝(3)/钱包(1)，`OrderNoGenerator` 生成单号，回调 `wxNotify/zfbNotify` 更新订单 + 写 `income_pay_detail`；退款 `PayService.wxRefund / zfbRefund`（退款后订单 status→4） | 没有"服务费/杂费"这种通用收费单类型，`payType` 需扩展；服务费金额计算（10% 或阶梯）要后端算 |
| **后台审核（通过/驳回）** | 有成熟先例：`income_pay_detail.tradeStatus`（1审核中/2审核失败/3已完成…）、教练 `coach.approveStatus/approveTime/approveResult`、后台 `SysIncomePayDetailController.updateTradeStatus` | 转让没有自己的审核表与审核界面，需新建 |
| **通知推送** | `system_msg` 表（站内信）+ 友盟 `UMengPush.sendAndroidUnicast/sendIOSUnicast` | 缺转让各节点的通知模板（发起/通过/驳回/待确认/超时/生效） |
| **拉黑会员不可转让 / 账号冻结** | 会员"封禁" = `user_info.auditStatus=2`（`BaseController.getUserVo` 校验，命中抛 `ERROR_USER_IS_LOCK`）；`device.status` 2–5 为各类禁用 | 无独立"黑名单"表/拉黑原因；转让前置校验需新增 |
| **门店等级（高级可向低级转）** | 多门店结构 `store` + `store_address`；`SysStoreEntity` **无 grade/level/type 字段** | 必须给门店加"等级"字段 + 后台维护 + 转让时校验"源等级 ≥ 目标等级" |
| **月卡停卡 12 次** | **完全没有停卡功能**（无停卡/冻结/暂停任何代码） | 全新：停卡申请、次数限制、有效期顺延 |
| **实时购买人数 + 每 100 人加 X 元** | `store.currentNum` 是"场内实时人数"，**不是购买人数**；`fit_card` 有价格阶梯但无"按人数动态涨价" | 全新：购买人数统计/展示 + 动态定价规则（可配置） |
| **私教课时/储值/运动档案不随权益转移** | 三者均 `userId` 强绑定：`private_class_order`(课时 `buyCount`)、`user_wallet`(`money` 储值)、`sport_record`(运动档案) | 设计上**天然满足**——只要转让只动"权益"不动这三张表即可，无需迁移 |

> 关键判断：**你系统里"VIP 权益"目前并不存在为一个独立对象**。这是整套方案最核心的建模问题，直接决定工作量与改动面，列为第 4 节头号决策点。

## 4. 关键决策点

### 4.1 已确认（2026-06-26 拍板）

| 编号 | 决策 | 结论 |
|---|---|---|
| **D-1** | "VIP 权益"建模 | **✅ 新建轻量权益层**：新建 `vip_benefit` 表，由会员持有的会员卡派生出一份独立权益（自带 1 年有效期、原始售价、可转让状态），与卡所有权分离。满足"卡与权益相互独立、只转权益"，复用现有买卡链路，改动可控。 |
| **D-4** | 转让服务费规则 | **✅ 可配置阶梯**：后台可配规则——首次转让可免费 + 第 N 次起按比例（默认 10%）或定额收取。同时覆盖规则文档两种说法。 |
| **D-3** | 受让人"补缴费用" | **✅ 固定工本费/确认费（可为 0）**：受让人确认时只付一笔后台可配的固定小额费用，可设为 0（即纯点确认）。不做"补差额"。 |
| **D-5** | 停卡规则 | **✅ 年卡每自然月可停 1 次、全年共 12 次**；停卡期间**有效期顺延**、停卡时不可入场；**无需后台审核**（会员自助）。 |

### 4.2 按推荐默认值定下（次要点，如无异议即按此实现）

| 编号 | 问题 | 默认结论 |
|---|---|---|
| **D-2** | 权益到期"失效 vs 顺延"（文档表述互斥） | 到期**失效**（与现有卡到期逻辑一致）；"顺延"只用于停卡期间的有效期顺延。 |
| **D-6** | 门店等级如何定义 | `store` 加 `store_level` 整数字段（数字越大等级越高），后台维护；转让校验"转让人门店等级 ≥ 受让人当前门店等级"。**档位数量待你给（默认先支持任意整数分级）**。 |
| **D-7** | 首发动态涨价涨谁 | 涨**该权益/卡的购买价**；规则可配（起涨基数、每多少人、每档加多少、是否封顶），后台不可手改实时人数。 |
| **D-8** | 文档格式 | Markdown 落在仓库 `需求文档/`。需要给门店/非技术方看时，可另导出 `.docx`。 |

## 6. 接口与流程设计

### 6.1 接口清单（移动端 `api` + 后台 `sys`）

移动端（`BaseController.getUserId/getUserVo` 取身份）：
- `POST /api/vipTransfer/quote` 试算服务费（按权益原价、转让次数、规则）
- `POST /api/vipTransfer/apply` 发起转让（校验前置条件→建转让单→生成服务费支付单）
- `GET  /api/vipTransfer/myList` 我的转让/受让记录
- `POST /api/vipTransfer/withdraw` 转让人撤回（通过后撤回不退费）
- `POST /api/vipTransfer/confirm` 受让人确认接收（→生成补缴支付单）
- `POST /api/vipTransfer/reject` 受让人拒绝
- 服务费/补缴支付复用现有 `PayController` 下单 + `wxNotify/zfbNotify` 回调
- `POST /api/cardPause/apply`、`GET /api/cardPause/list` 停卡（R-06）
- 权益购买页展示购买人数 + 动态价：在现有 `FitCardController.detail` 返回里补字段（R-07）

后台（`sys`，Shiro 鉴权）：
- `GET  /sys/vipTransfer/list` 转让申请列表（待审核/已通过/已驳回/已生效）
- `POST /sys/vipTransfer/audit` 审核通过/驳回（驳回触发退费 + 通知）
- `GET  /sys/store/...` 门店等级维护（在现有 `SysStoreController` 加字段）
- 服务费规则、动态涨价规则配置项（仿 `SysSetController.memberLevel` 在线配置）

### 6.2 转让全流程 + 状态机

`vip_benefit_transfer.status` 流转：

```
[10 待付服务费]
   └─转让人支付服务费成功→ [20 待门店审核]
        ├─审核驳回→ [31 已驳回] →全额退服务费→结束
        └─审核通过→ [40 待受让人确认]（推送受让人）
             ├─受让人拒绝/超时未确认→ [51/52 已拒绝/已超时] →全额退服务费→结束
             ├─转让人撤回（通过后）→ [60 已撤回] →服务费不退→结束
             └─受让人确认并补缴成功→ [70 已生效]
                   → 过户事务：
                      · vip_benefit.user_id 改为受让人，transferable=0，transfer_count+1
                      · 原持有人该权益 status=已转出失效（立即丧失）
                      · expire_time = 缴费当日 + 1 年（重置）
                      · 写 income_pay_detail（服务费收入 + 补缴收入，anotherId 记对方）
                      · 私教课时/储值/运动档案不动（留转让人）
                      · 推送双方"转让完成"
```

事务边界：过户那步必须单事务（命中 `api.service.impl` 事务切面，REQUIRED）。支付回调要做幂等（防重复过户），沿用订单号去重。

### 6.3 前置校验规则集（发起 + 审核两道关，R-08/17）

拒绝转让的条件（任一命中即拦截/驳回）：权益已过期/不在有效期；账号被封禁（`auditStatus=2`）或在黑名单；存在欠费；已办理退卡；重大违规记录；该权益 `transferable=0`（受让所得或已转过）；门店等级不满足"源≥目标"。

### 6.4 定时任务（复用现有 `api.schedule` + `spring-mvc.xml` 注册）

- 受让人确认超时扫描 → 自动置"已超时" + 退服务费（可挂到现有每小时 `UpdateOrderStatusTask` 节奏）。
- 权益到期失效（R-03）→ 仿 `UpdateClassStatusTask`。
- 停卡到期自动恢复 + 有效期顺延。

## 8. 完成态 / 验收清单（做成后长什么样）

- 会员在小程序"我的权益"里能看到自己可转让的 VIP 权益，点"转让"→填受让人→看到服务费试算→支付服务费→提交申请。
- 后台"权益转让审核"列表能看到待审申请，1–2 工作日内点通过/驳回；驳回自动退服务费并通知双方。
- 通过后受让人收到推送，在小程序确认并完成补缴；支付成功的**那一刻**：转让人这份权益立刻失效，受让人立刻拥有全套权益且有效期重置为 1 年；双方收到"转让完成"通知。
- 受让人无法再次转让该权益、无法退费；入场仍走本人账号。转让人的私教课时/储值/运动档案分文未动。
- 黑名单/欠费/过期/退卡/违规/门店等级不符的会员，发起或审核时被明确拦截并提示原因。
- 月卡会员可在小程序申请停卡，受 12 次/有效期顺延规则约束。
- 权益购买页公开展示实时购买人数；达到配置人数阈值自动按规则涨价，后台不可手改人数。
- 门店后台可维护门店等级；转让严格遵守"高级门店权益→低级门店"单向规则。

**不做的事（明确划界）**：不支持权益拆分转让、不支持私下交易撮合、不介入双方私下钱款纠纷、不转移卡所有权与个人资产。

## 9. 工作量与风险提示

- 工作量集中在三块：①权益层 + 转让状态机（核心、最大）；②停卡（中等、独立）；③门店等级/动态定价/购买人数（小、独立，可先行）。
- 主要风险：**D-1 建模决策**直接决定改动面，必须先定；过户事务与支付回调的**幂等与并发**（防重复过户/重复退费）；退费依赖微信/支付宝退款的可用性与对账。
- 建议落地顺序：先定决策点 → 先做"门店等级 + 购买人数/动态定价 + 停卡"这些独立小项（风险低、可独立验收）→ 再做权益层与转让主流程。
